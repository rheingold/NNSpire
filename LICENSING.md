## 1. Core license split

| Component | License | Rationale |
|---|---|---|
| `NNSpire-core` (engine library) | **GNU LGPL v3** | Proprietary plugins may dynamically link against the core without GPL infection. Anyone modifying the core must publish those modifications. |
| `NNSpire/plugin-api/` (SDK headers + TrustVerifier) | **GNU LGPL v3** | Same — plugin authors link against these headers. |
| `NNSpire/python-bridge/` (pybind11 module) | **GNU LGPL v3** | Bridge to core; same terms. |
| `NNSpire/app/` (Qt application shell) | **GNU GPL v3** | The application is not a library; GPL prevents proprietary forks of the Studio UI without source disclosure. |
| `NNSpire/backends/` (CPU, CUDA, Quantum) | **GNU LGPL v3** | Backend plugins are linkable libraries. |
| `NNSpire/plugins/` (built-in reference plugins) | **MIT** | Reference plugins are examples; widest possible use for learning and scaffolding. |
| `NNSpire-sign` CLI tool | **GNU GPL v3** | Standalone executable. |
| AI Standards KB (`ai-standards-kb/`) | **CC BY 4.0** | Documentation / knowledge base. |

Full license texts are in `LICENSES/` at the project root.

### What the LGPL split means in practice

A third party building a commercial plugin:
- May dynamically link their closed-source plugin `.dll`/`.so` against `NNSpire-core` and `NNSpire-plugin-api` — this is **explicitly permitted** by LGPL v3 § 4.
- Must not statically link `NNSpire-core` into their plugin and ship it without making the core source available — static linking would require GPL/LGPL disclosure.
- The recommended packaging (dynamic link, separate shared library) is the scaffold generated by `NNSpire-sign keygen`, so compliance is automatic.

### What GPL on the app shell means

Anyone who distributes a modified version of the Studio application (`NNSpire/app/`) must:
- Publish their modifications under GPL v3.
- This includes forks of the Studio that disable the trust verification system — **disabling the trust chain and distributing the result is a GPL violation**.

## 2. Plugin charter

### 2.1 Free and permanently open (project commitment)

The following capabilities are declared `FREE` in `NNSpire/core/features/FeatureFlags.h` and are committed to remain `FREE` in all future versions of NNSpire:

| Category | Scope |
|---|---|
| **Learning & wizard mode** | All guided walkthroughs, wizard steps, educational visualizations |
| **Visualization** | Weight heatmaps, neuron viewer, loss charts, activation histograms, topology diagram |
| **Full pipeline** | All input adapters, tokenization, context/RAG, execution, output adapters |
| **Standard runner connectors** | Triton, TF Serving, KServe, ONNX Runtime embedded, OpenAI-compatible REST |
| **KB help system** | In-app help, context-sensitive KB popups, wizard KB integration |
| **All standard export formats** | ONNX, TorchScript, `.nns`, `.nnsr` |
| **Plugin SDK** | The C ABI, Python bridge, scaffolding tools, `NNSpire-sign` CLI |
| **Trust verification itself** | The signing/verification mechanism is open; the *issuance service* is commercial |

### 2.2 Commercial plugin economy

The intended primary revenue stream for NNSpire is the **commercial plugin ecosystem**:

- Plugin authors pay a registration fee to obtain a Vendor Signing Certificate.
- The project owner holds the only private key capable of issuing Vendor certs.
- Plugin authors are free to price, license, distribute, and DRM-protect their own plugins in any way they choose, within the constraints that:
  1. The source code must be disclosed (even if DRM-protected against execution) — this is an explicit policy of the NNSpire registry.
  2. The plugin must pass the dual-language compliance verification (C++ + Python both present and behaviourally equivalent).
  3. The plugin manifest must accurately declare all capabilities.

### 2.3 Community plugin economy

Open-source or free plugins may obtain a Community Signing Certificate at no cost via `NNSpire-sign submit --type community`. Requirements:
- Public source code repository URL provided at registration.
- Source must be published under an OSI-approved license.
- Dual-language compliance required (same as commercial).
- Certificate issued automatically (no human review) after repository check.

### 2.4 Enterprise self-signing

Large organisations may obtain an Enterprise Intermediate CA certificate (see `TRUST-ARCHITECTURE.md` §6). This allows them to:
- Self-sign all internal plugins without per-plugin registration.
- Deploy internal plugins to employee machines via the Trust Update Package mechanism.
- Never expose internal plugin code to the public registry.

Cost: one-time enterprise CA issuance fee (commercial arrangement with project owner). Renewal on cert expiry.

## 3. "Pro" tier policy

A `PRO` or `ENTERPRISE` feature tier is **not currently planned** and must not be introduced casually.

Rules for any future consideration of a paid tier:
1. Learning, visualization, and full pipeline features may **never** be gated behind a paid tier.
2. Any proposed Pro feature must be documented as an issue, discussed publicly, and only introduced after a community comment period.
3. The change from `Tier::FREE` to `Tier::PRO` in `FeatureFlags.h` constitutes a documented, version-controlled decision with a required rationale comment.
4. If a Pro tier is ever introduced, a perpetual free legacy version must be maintained for all features that were previously free.

This policy is recorded here as a public commitment, enforceable by the open-source license terms (GPL/LGPL require that source modifications be disclosed, including feature flag changes).

## 5. Contributor license agreement (CLA)

All contributors to `NNSpire-core`, `NNSpire/app/`, and `NNSpire/plugin-api/` must sign a CLA (to be drafted) granting the project owner:
- The right to dual-license the contribution (e.g., release a future commercial SDK build).
- Confirmation that the contribution is the contributor's own original work.
- Confirmation that the contribution is **not encumbered by third-party patents** that would restrict NNSpire's ability to use, modify, or distribute it under the project's chosen licenses.

The CLA does not affect the open-source license of the project — contributions remain LGPL/GPL. It only grants additional rights to the project owner to enable the commercial plugin economy described above.

A lightweight CLA process (GitHub CLA bot or DCO sign-off) will be decided before the first public contribution is accepted.
